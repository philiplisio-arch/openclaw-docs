# OPENCLAW-ADV-015: Semantic Validation Gap

## 1. CONTEXT

D17 (2026-06-06) contains a confirmed cross-source citation misbinding (D-FB-008):
ET Bullet 2 states the European Commission declared EU-China trade
"not sustainable" and that UK FM Yvette Cooper is visiting China.
The cited source (res_0a5837f3bb9f, h5.article.smbae.cn) has a snippet
entirely about US-China tariffs. Zero mention of the EC, Yvette Cooper,
or the UK appears in any of the 12 retrieval package snippets.

The validator returned GREEN 16/16/0. This is correct behaviour — the
validator checks that cited result_ids exist in the retrieval package and
that citation syntax is valid. It has never been designed to verify that
a source's content supports the specific claim made against it.

The "fabrication rate 0%" conclusion from prior sessions measured the
wrong thing for this failure mode. It confirmed all cited result_ids are
real sources. It did not confirm cited sources support their claims.

---

## 1A. PRECISE FAILURE MODE NAME (consultant refinement, 2026-06-06)

The failure identified in D17 should not be described as "fabrication" in
the prior sense (agent inventing result_ids). The precise term is:

**Cross-source citation misbinding** — the agent absorbs real facts from
one or more higher-authority retrieval results, combines them into a
single bullet, and cites a different lower-authority source that does not
support the specific claim. The cited result_id is real and present in
the retrieval package. The grounding is false.

This is more subtle than fabricated result_ids (Issue #43, resolved
2026-05-07) and more dangerous: the citation passes syntax validation
and the fact may be real, but the source link is false.

The system is currently:
- **Citation-structure safe** — all cited result_ids exist in the package ✓
- **Not yet client-factual-grounding safe** — cited source content does
  not guarantee it supports the specific claim ✗

---

## 2. NATURE OF THE GAP

The current validation model guarantees:
- Every result_id cited by the agent exists in the retrieval package ✓
- Citation syntax matches the locked format ✓

The current validation model does NOT guarantee:
- The cited source's content supports the specific claim ✗
- The agent did not draw on training knowledge and attach a result_id ✗
- The claim would survive source verification by the client ✗

This gap allows a specific failure mode to pass undetected: the agent
generates a factually plausible claim from training knowledge, selects
a topically adjacent result_id from the retrieval package, and the
validator confirms the result_id exists. The claim is delivered as
source-backed when it is not.

---

## 3. FIX OPTIONS

### Option A — Snippet Grounding Check (prompt layer)
Instruct the agent to only cite sources whose snippet directly supports
the claim. PROHIBITED — orchestrator spec explicitly prohibits
prompt-based filtering. Not viable.

### Option B — Post-Agent Snippet Alignment Check (approved for design)
After agent output but before scrubber, run a deterministic check: for
each cited result_id, verify that at least one keyword from the claim
appears in the source's snippet. Flag misaligned citations as WARN in
validation_result.json.

Mechanism: extract noun phrases from each bullet; check for overlap with
corresponding source snippets; if overlap below threshold, add
`citation_alignment_warning` to validation_result.json.

Risk: MODERATE-HIGH. Keyword matching is a weak proxy for semantic
alignment, especially across languages (Chinese snippets vs. English
claims). Threshold calibration is non-trivial. Requires careful spec
design before any Claude Code implementation. Operator approves spec
before implementation begins.

### Option C — Retrieve-and-Verify Layer (browser retrieval)
Use Browser Retrieval to fetch full article text for cited sources and
verify claim grounding against full content rather than snippets.
OUT OF SCOPE — Browser Retrieval Phase 2 not authorized.

### Option D — Human Spot-Check Protocol (approved — immediate)
Operator spot-checks one randomly selected cited source per delivery
against its claim. If the source does not support the claim, the
delivery is held and the issue is escalated. Manual but requires no
implementation and provides immediate coverage. Added to
OPENCLAW_PHASE_D_OPERATOR_REVIEW_PROCEDURE.md as standing Step 2A.

---

## 4. FABRICATION RATE LANGUAGE CORRECTION

All references to "fabrication rate 0%" in system documents are to be
understood as: zero result_id syntax failures detected by the validator.
They do not assert semantic grounding of claims against source content.
This distinction is now recorded in system documents.

The validator GREEN status remains meaningful and accurate for what it
measures. The semantic gap is a separate, additive concern addressed by
Option D (immediate) and Option B (medium-term).

1. Option D (manual spot-check) — APPROVED; added to review procedure
2. Option B design work — APPROVED; spec to be drafted before Claude Code
3. Fabrication rate language — APPROVED for correction in system docs
