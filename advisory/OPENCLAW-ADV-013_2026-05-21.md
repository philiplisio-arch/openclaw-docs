**Track:** Research-only parallel track (per OPENCLAW-OPS-001 v2.6, Section 2)  
**Scope:** Standalone article fetch capability — zero contact with production pipeline  
**Installation path:** `/root/openclaw_phase7/browser_retrieval/`  
**Advisory status:** Findings only. Phase 2 integration requires separate operator decision.

## 1. Installation Summary

Playwright 1.60.0 and headless Chromium (Chrome for Testing 148.0.7778.96) were
installed on the VPS at `/root/.cache/ms-playwright/`. All system dependencies
were resolved via `playwright install-deps chromium`.

**One install note:** On Ubuntu 24.04, `pip install playwright` requires the
`--break-system-packages` flag due to PEP 668 external environment protections.
The `playwright` CLI is not on PATH by default after pip install; `python3 -m playwright`
is the reliable invocation. Both are one-time setup notes with no ongoing impact.

## 3. Findings

### Finding 1 — Western news sites: fully functional

The CNBC article extracted cleanly via `[role='main']` with 631 words and an
11.7s fetch time. Content quality was verified manually — full article body with
no nav/ad contamination. The semantic selector priority list works as intended
for standard Western news sites.

**Assessment:** Brave result URLs from Western publishers are retrieval-ready.
No further development needed for this category.

---

### Finding 2 — Chinese mobile sites: content present, word counter required fix

The Sina Finance mobile page (`finance.sina.cn`) initially appeared to return
only 49 words across two extraction attempts (v1.0 and v1.1 with JS-wait
improvements). Investigation revealed the content was fully present — the
failure was in the word counter, not the extraction.

**Root cause:** `count_words()` used Python's `str.split()`, which splits on
whitespace. Chinese text has no spaces between words, so a 1,588-word article
returned a split count of 49 (only date stamps, UI labels, and numbers
registered as tokens).

**Fix applied (v1.1):** `count_words()` now detects CJK character ranges
(U+4E00–U+9FFF and adjacent blocks), counts CJK characters divided by 2
(average Chinese word length), and adds space-separated token count from
non-CJK portions. Post-fix word count: **1,588** — consistent with the
article's actual content volume (a detailed Ministry of Commerce Q&A on
US-China trade negotiations).

**Residual note:** The `body (fallback)` selector fired rather than a
Chinese-specific selector. The article content was still extracted correctly
via the fallback. Adding a targeted Sina mobile selector (`#artibody`,
`.art_content`) would improve selector specificity but is not required for
functional extraction.

---

### Finding 3 — Hard paywalls: correctly detected and reported

WSJ returned HTTP 401 before serving any content. The script correctly:
- Logged `status=FAIL` with `error=HTTP_401`
- Exited with non-zero status
- Wrote nothing to the output file
- Completed in 0.8s (fast fail — no wasted browser time)

Soft paywall detection (word count < 200 + keyword matching) was not triggered
in this test because WSJ blocks at the HTTP layer. That path remains untested
and should be validated against a soft-paywall site (e.g., NYT metered access)
before Phase 2 integration planning.

## 4. Script Capability Summary

`fetch_article_text.py` as of v1.1 is capable of:

- Fetching and extracting full article text from standard Western news sites
- Fetching and extracting full article text from Chinese mobile/JS-rendered sites
- Correctly counting words in both Latin and CJK scripts
- Detecting and reporting hard paywall blocks (HTTP 4xx)
- Detecting potential soft paywalls via word count + keyword signals
- Logging fetch status (URL, words, selector, elapsed time) to stderr for
  downstream log analysis
- Writing extracted text to file or stdout for downstream consumption

It is a standalone research tool. It has no imports from or connections to
the production pipeline.

the Phase 2 integration design:

## 6. Phase 2 Integration — Advisory Only

**This section is advisory. Phase 2 integration is explicitly OUT OF SCOPE per
OPENCLAW-OPS-001 v2.6 and requires a separate operator decision.**

Phase 1 demonstrates that headless Chromium retrieval is viable for both
Western and Chinese source URLs present in current retrieval packages. If the
operator decides to proceed to Phase 2, the primary integration design question
is the handoff point: whether the browser fetcher supplements the existing
retrieval layer (fetching full text for URLs already retrieved by Brave/Baidu)
or replaces it for specific publisher categories.

That is an architectural decision outside the scope of this advisory.
